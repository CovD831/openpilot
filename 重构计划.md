# OpenPilot 重构计划

## 1. 重构目标

本计划以 `instructions/` 下的设计文档为最高优先级，用于指导后续逐步重构 `Code/` 中的现有实现。重构目标不是一次性推倒重写，而是在保持现有 CLI、测试和运行链路可用的前提下，将代码逐步收敛到明确的 Module / Agent / Tool / Utils / Memory / Autonomous Iteration 架构。

核心目标如下：

- 以 `src/{module}` 作为模块边界，每个模块负责一组稳定能力。
- Agent 由固定 prompt、agent functions 和 tool calling 组成，并通过清晰的公开函数接收输入、输出结果。
- `tools/` 只承载有标准输入输出协议、可被任意 Agent 调用的公共工具。
- `utils/` 只承载无 LLM、无 ToolDefinition 协议、低副作用的公共函数。
- Memory Module 提供短期上下文、长期记忆、项目结构检索和虚拟环境上下文能力。
- Autonomous Iteration Module 提供稳定的自主迭代流水线：Context Loader -> Goal Maker -> Task Designer -> Task Decomposer -> Task Executor。
- 每个 module、agent、tool 和关键函数都输出结构化日志，日志事件能够区分来源、阶段、输入摘要、输出摘要、错误和耗时。
- 优先保证兼容性：旧 import、CLI 入口、测试和现有数据文件在重构过程中不能无计划失效。

## 2. 当前状态判断

当前代码已经具备现代 Agent + Tool 架构雏形，但与设计文档仍存在一些结构偏差：

- `Code/src/execution/intelligent_autopilot.py` 体量过大，集中承担了执行编排、项目迭代、工具调用、UI 更新、错误处理和日志等多种职责。
- `Code/src/agents/` 当前是全局 Agent 池，后续应逐步向 `src/{module}/{agent}` 的模块内 Agent 组织方式靠拢，同时保留兼容导入层。
- `Code/src/tools/` 已有标准 ToolDefinition、ToolRegistry、ToolExecutor 和若干内置工具，但工具输入输出校验、权限、审计日志和失败恢复策略需要统一验收口径。
- `Code/src/utils/` 基本承担通用函数职责，但后续需要继续清理任何含 LLM、Tool 协议或业务编排语义的代码。
- `Code/src/memory/` 已有 `MemoryStore`、`ShortMemory`、`MemoryVault` 等基础能力，但设计文档中的 Project Manager/sketch、Memory Graph recall 和 Virtual Environment Manager 还需要进一步对齐。
- `Code/src/agents/iterative_improvement.py` 已经实现 Autonomous Iteration 雏形，但应按 Context Loader、Goal Maker、Task Designer、Task Decomposer、Task Executor 拆出更明确的 agent/function 边界。

## 3. 分阶段实施计划

### 阶段 0：基线冻结与测试盘点

目标：

- 在任何结构性改动前确认当前项目的可运行基线。
- 记录现有测试、CLI 和关键运行链路的真实状态。

改动范围：

- 不做业务代码重构。
- 补充或更新项目结构说明、重构检查清单和测试命令记录。
- 明确现有入口：`openpilot = ui.cli:main`。

执行步骤：

1. 在 `Code/` 下运行现有测试：`pytest`。
2. 验证 CLI 配置入口：`openpilot config check`。
3. 盘点当前核心链路：
   - `ui.cli`
   - `ui.enhanced_cli`
   - `execution.IntelligentAutopilot`
   - `agents.TaskDecomposer`
   - `tools.ToolOrchestrator`
   - `tools.ToolExecutor`
   - built-in tools
   - `agents.AutonomousIterationAgent`
   - `memory`
   - `core.OpenPilotLogger`
4. 记录已知失败、跳过项和需要后续修复的测试缺口。

风险：

- 现有测试可能依赖本地环境或 LLM 配置，不能把环境失败误判为重构失败。

验收标准：

- 有明确的基线测试结果。
- 知道哪些失败是现状，哪些失败是后续重构引入的回归。

### 阶段 1：公共协议与日志规范收敛

目标：

- 先统一公共协议，再移动实现细节。
- 为后续模块拆分提供稳定的 contracts 和日志格式。

改动范围：

- Tool 协议、Agent 执行上下文、Memory 数据模型、Autonomous Iteration 结果模型。
- `core.openpilot_log` 或新的日志辅助层。

执行步骤：

1. 梳理公共类型归属：
   - Agent 类型保留在 `agents/*_models.py` 或迁入对应 module 的 models。
   - Tool 类型保留在 `tools/tool_models.py` 和 `tools/tool_orchestration_models.py`。
   - Memory 类型保留在 `memory/memory_models.py`。
   - Execution 类型保留在 `execution/*_models.py`。
2. 定义统一日志事件字段：
   - `timestamp`
   - `session_id`
   - `turn_id`
   - `source_type`
   - `source_name`
   - `phase`
   - `event_type`
   - `input_summary`
   - `output_summary`
   - `duration_ms`
   - `success`
   - `error`
   - `metadata`
3. 为 module、agent、tool、function 建立轻量日志 helper，避免每处手写不一致 JSON。
4. 保留 `OpenPilotLogger.log_event(...)` 的兼容接口，新增结构化 helper 通过它落盘。

风险：

- 日志字段变更可能影响测试或既有日志解析脚本。
- 过早修改模型字段可能引入大量连锁改动。

验收标准：

- JSONL 日志仍可被逐行解析。
- 旧的 `log_event` 调用仍然可用。
- 新日志能够明确区分 module、agent、tool 和 function 来源。

### 阶段 2：Tools / Utils 边界清理

目标：

- 让 `tools/` 和 `utils/` 的边界严格符合设计文档。
- 工具具备稳定输入、输出、权限、失败模式和审计语义。

改动范围：

- `Code/src/tools/`
- `Code/src/utils/`
- 与工具注册和执行相关的测试。

执行步骤：

1. 盘点所有工具，确认每个工具都有：
   - `ToolDefinition`
   - 输入 schema
   - 输出 schema
   - permission level
   - failure modes
   - executor 函数
   - audit 设置
2. 对设计文档指定的公共工具建立明确入口：
   - File Reader
   - File Writer
   - Command Executor
   - Embedder
3. 清理边界：
   - 需要 LLM 或 Agent 调用协议的能力放入 `tools/`。
   - 纯文本、JSON、diff、缓存、并发、树形展示等通用能力放入 `utils/`。
4. 为 ToolExecutor 增强统一校验：
   - 缺失必填参数时返回结构化错误。
   - 输出字段不符合定义时记录 validation warning。
   - 工具失败时附带 failure mode 和 recovery hint。
5. 保持 `register_builtin_tools(...)` 作为内置工具注册入口。

风险：

- 工具输出格式收紧可能暴露历史调用中未显式处理的错误。
- 过度抽象可能让简单工具变复杂。

验收标准：

- 所有内置工具可注册、可查询、可执行。
- 文件读取、文件写入、命令执行、embedding 相关能力有明确 Tool 或 Utils 归属。
- 工具执行失败时返回结构化错误而不是裸异常。

### 阶段 3：Memory Module 对齐

目标：

- 将 Memory Module 对齐到设计文档中的 Context Manager、Memory Vault、Project Manager 和 Virtual Environment Manager。
- Memory 不只是 JSONL 存储，而是能为 Agent 提供上下文、偏好、项目文件和环境信息。

改动范围：

- `Code/src/memory/`
- 现有 memory tools 和 env tools。
- 与 project state、environment context 相关的工具。

执行步骤：

1. Context Manager：
   - 用户原话完整保留。
   - Agent 历史通过 compressor 压缩。
   - 提供 `output_context` 风格的公开函数。
2. Memory Vault：
   - 保留现有 `MemoryRecord` 兼容字段。
   - 明确 Memory Node 与 Memory Link 概念。
   - 将 `embedding`、`related_memory_ids`、`recall_frequency` 映射到 graph recall 语义。
   - 补齐 confidence evaluation 和 remind/search 的返回协议。
3. Project Manager：
   - 为项目目录维护 `sketch.json` 的设计落点。
   - 每个 sketch 记录文件名和文件功能描述的 embedding 或可延迟生成占位。
   - 提供 update/search 两类公开函数。
4. Virtual Environment Manager：
   - 明确环境创建、包安装命令生成、环境上下文读取的边界。
   - 将已有 `env_tools` 中可复用逻辑收敛到 Memory Module 或其工具包装层。
5. Memory 与 Tools 的边界：
   - Memory Module 实现状态管理。
   - Agent 可调用的入口通过标准 Tool 暴露。

风险：

- `sketch.json` 会引入新运行时文件，需要避免污染源码和测试。
- embedding 可能依赖外部模型，应提供无 embedding 的降级路径。

验收标准：

- Memory 查询、保存、更新 usage 的现有行为不回归。
- Context 输出能包含用户历史、压缩 Agent 历史、相关文件、相关记忆和环境信息。
- Project Manager search 能根据查询返回相关文件路径。
- 环境上下文工具能在无外部 LLM 的情况下返回基础环境信息。

### 阶段 4：Autonomous Iteration Module 对齐

目标：

- 将自主迭代逻辑收敛为设计文档指定的稳定流水线。
- 降低 `IntelligentAutopilot` 的体量和职责耦合。

改动范围：

- `Code/src/execution/intelligent_autopilot.py`
- `Code/src/agents/iterative_improvement.py`
- `Code/src/agents/task_decomposer.py`
- 与项目评估、工具执行、UI 进度相关的适配层。

执行步骤：

1. Context Loader：
   - 从 Memory Module 读取系统 prompt、压缩对话、相关文件、相关记忆和虚拟环境信息。
   - 输出统一 context object，供后续 agent 使用。
2. Goal Maker：
   - 接收 context，生成项目改进方向。
   - 输出结构化 improvement goals。
3. Task Designer：
   - 接收 context 和 goal，生成具体 tasks。
   - 输出包含验收标准、风险和优先级的任务对象。
4. Task Decomposer：
   - 接收 context 和 task。
   - 递归拆分直到任务足够简单。
   - 保留 Task Difficulty Evaluator 作为 agent function。
5. Task Executor：
   - 接收 context 和 subtask。
   - 通过 ToolOrchestrator / ToolExecutor 执行。
   - 返回结构化执行结果、文件变更摘要、失败原因和下一步建议。
6. 拆分 `IntelligentAutopilot`：
   - 保留外部 `execute(goal, context=None)` 兼容入口。
   - 将上下文加载、项目迭代、工具执行回调、UI 适配、日志适配拆为内部组件。
   - 避免一次性移动全部代码，优先抽取低风险纯辅助函数。

风险：

- 主执行路径复杂，拆分时容易破坏增强 UI、日志或测试 mock。
- Autonomous Iteration 涉及 LLM、文件写入、项目验证，必须小步提交和测试。

验收标准：

- `IntelligentAutopilot.execute(...)` 的外部行为兼容。
- `/autopilot` 主路径仍能调用任务分解、工具执行和项目迭代。
- Autonomous Iteration 每个阶段都有独立日志事件。
- 失败时能明确定位失败阶段。

### 阶段 5：兼容层、文档、测试和废弃代码清理

目标：

- 完成结构迁移后的收尾工作，移除已无用或冲突的旧路径。
- 确保后续开发者能根据文档理解边界。

改动范围：

- `README.md`
- `PROJECT_STRUCTURE.md`
- 测试目录
- 兼容 import 层
- 已废弃文件和重复实现。

执行步骤：

1. 更新 README 中的当前架构图和运行链路。
2. 更新 `PROJECT_STRUCTURE.md`，将实际目录与重构后的模块边界同步。
3. 为核心协议补充测试：
   - ToolDefinition 注册与执行。
   - Memory context 输出。
   - Autonomous Iteration 阶段事件。
   - CLI 入口可用性。
4. 保留必要兼容层，并为计划废弃的 import 增加清晰注释。
5. 删除或合并与新设计冲突的重复实现。

风险：

- 删除旧兼容路径可能破坏外部调用。
- 文档与代码不同步会降低后续重构效率。

验收标准：

- README、PROJECT_STRUCTURE 和实际代码一致。
- 测试覆盖重构后的关键模块边界。
- 没有已知重复实现与新设计冲突。

| --- | --- | --- | --- | --- |

| P0 | `Code/src/tools/` | Tool 协议存在但校验和日志口径不完全统一 | 统一 ToolDefinition、输入输出校验、失败模式和审计事件 | 所有内置工具可注册、可执行、可结构化失败 |
| P1 | `Code/src/memory/` | Memory Graph、Project Manager/sketch 和环境上下文尚未完整对齐 | 分别补齐 Context Manager、Memory Vault、Project Manager、Virtual Environment Manager | Agent 能通过 Memory 获取上下文、记忆、文件和环境信息 |
| P1 | `Code/src/agents/iterative_improvement.py` | 自主迭代有雏形，但阶段边界不够清晰 | 拆出 Context Loader、Goal Maker、Task Designer、Task Decomposer、Task Executor | 每个阶段可单测、可日志追踪、可独立失败 |

## 5. 测试与验收策略

每个阶段开始前：

```bash
cd /Users/yanning/Projects/openpilot/Code
pytest
```

每个阶段完成后至少验证：

- `openpilot config check` 不回归。
- `Code/tests/` 现有测试通过，或记录与环境相关的已知失败。
- Tool 注册、文件读写、命令执行、Memory 读写、Autonomous Iteration 主路径仍可用。
- 新增或调整日志字段不破坏 JSONL 可解析性。
- 所有新增结构保留向后兼容入口，除非某一步明确声明并验证移除。

推荐新增测试场景：

- Tool 输入缺失、类型错误、执行失败时返回结构化错误。
- Memory context 能拼接用户原话、压缩 Agent 历史、相关记忆和环境摘要。
- Project Manager 能生成或更新目录 sketch，并通过 query 找回相关文件。
- Autonomous Iteration 在 Goal Maker、Task Designer、Task Executor 任一阶段失败时能记录阶段化错误。
- `IntelligentAutopilot.execute(...)` 在 mock LLM 和 mock tool executor 下保持返回结构稳定。

## 6. 执行原则

- 小步重构，每一步都保持测试和 CLI 尽量可运行。
- 优先收敛协议和边界，再移动文件和目录。
- 保留兼容 import，避免一次性破坏调用方。
- 对运行时产物保持克制，`logs/`、`data/`、`sketch.json` 等文件要有清晰生成和忽略策略。
- 所有新公共接口都要写明输入、输出、失败模式和日志事件。
- 当现有代码与 `instructions/` 设计冲突时，以 `instructions/` 为准；如果为了兼容短期不能完全对齐，应在代码注释或文档中标出过渡策略。

## 7. 非目标

- 本轮计划不直接重写整个项目。
- 不一次性迁移全部目录结构。
- 不在没有测试保护的情况下删除兼容入口。
- 不把 LLM 调用、Tool 协议或业务编排逻辑塞进 `utils/`。
- 不让 Memory 的 embedding 能力成为基础功能的硬依赖。
