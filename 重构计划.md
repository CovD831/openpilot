# OpenPilot Agent Runtime 重构计划

## 1. 目标与原则

本计划基于 `/Users/yanning/Projects/openpilot/references/insights/agent_runtime_design_insights.md` 重写，目标是把 OpenPilot 的执行运行时重构为一个状态驱动、阶段式、证据优先、可验证、可解释的 Agent Runtime。

重构后的 OpenPilot 不应把工具调用当作固定流程或能力展示，而应把工具视为“补全当前决策所缺证据”的手段。Agent 应先基于已有状态推理，只有当继续判断需要真实环境信息、外部状态、执行结果或用户决策时，才触发工具、编辑、验证或询问用户。

核心原则：

- Phase Graph 优先于 Fixed Tool Graph：先规划阶段，再在阶段内动态选择工具。
- State 优先于上下文堆叠：所有工具结果都必须被吸收成显式 runtime state。
- Need 优先于 Tool：模型表达“需要什么”，Tool Router 决定“调用什么”。
- Evidence 优先于猜测：文件选择、编辑、验证、修复都必须有可追溯证据。
- Minimal Loop 优先于一次性大改：复杂任务拆成多个理解、计划、执行、验证、总结闭环。
- Verification 是写操作后的硬门禁：任何改变项目状态的动作后都必须验证。
- Explainability 是运行时能力：系统必须能解释每次工具调用、编辑和重新规划。
- Target Architecture 优先于兼容包袱：如果旧结构与新 Runtime 职责重复，删除旧结构或迁移调用方，不保留双轨实现。
- Single Source of Truth：阶段推进、工具选择、编辑授权、验证决策和报告生成只能各有一个权威入口。

## 激进重构边界

本计划采用目标架构优先原则。只要旧结构与新结构作用相同、职责重叠或会造成双轨维护，就应删除、合并或迁移，而不是保留兼容层。

允许删除或合并的旧结构包括：

- 直接让模型输出固定 `tool_calls` 的旧工具规划路径。
- 与 `AgentRuntimeController` 重复承担阶段推进、任务流转或执行汇总的旧 runner/helper。
- 与 `ToolRouter` 重复承担工具选择职责的 prompt 规则、临时映射函数或散落在 executor 中的工具选择逻辑。
- 与 `StateUpdater` 重复承担结果解析和状态更新的旧上下文拼接逻辑。
- 与 `EditGuard` / `RuntimeGuard` 重复承担权限、风险、预算、确认判断的旧 guard 或临时 if 分支。
- 与 `RuntimeVerifier` 重复承担验证命令选择、验证状态判断、失败恢复入口的旧验证路径。
- 与 `RuntimeReporter` 重复承担最终结果汇总、工具历史解释、风险说明的旧报告逻辑。

删除规则：

- 删除前先确认没有唯一能力只存在于旧结构中；如有唯一能力，迁移到新职责组件后删除旧入口。
- 删除旧结构时同步删除只为旧结构服务的 metadata、测试桩、prompt 文案和 README 描述。
- 不新增“兼容适配器”来维持旧 API；外部 CLI 也可以按新 Runtime 重新设计。
- 旧测试不再作为行为约束原样保留；应改写为验证新 Runtime 的行为。
- 对外返回结构可以发生破坏性调整，只要新返回结构能完整表达目标、状态、证据、修改、验证和风险。

## 2. 目标行为

面对任意项目任务，OpenPilot Runtime 应具备以下行为：

1. 判断当前阶段是否能靠已有状态继续推理。
2. 如不能，形成 `DecisionNeed`，说明缺什么信息、为什么缺、结果会影响什么决策。
3. 由 `ToolRouter` 根据 need、阶段、风险、预算和权限选择最小必要工具。
4. 工具执行前由 Guard 检查权限、风险、预算、确认需求和项目状态影响。
5. 工具执行后由 `StateUpdater` 把结果转化为事实、已解决问题、新问题、候选文件、风险变化、计划变化和下一步动作。
6. 需要编辑时先选择证据驱动的文件，再生成编辑计划，而不是让 Editor 自己选文件。
7. 每轮编辑只围绕一个清晰子目标，使用最小编辑集。
8. 写文件、删文件、生成代码、修改依赖、修改配置、执行可能改变项目状态的命令后，必须进入验证。
9. 验证失败后先诊断失败原因，再形成最小修复计划，不能盲目继续编辑。
10. 当关键假设被推翻、风险变化、计划不可执行、目标已完成或验证失败不符合原假设时，触发有边界的 Replanning。
11. 完成后报告修改、证据、验证结果、残余风险和停止原因。

## 3. Runtime 状态模型

新增或强化显式 runtime state，作为所有决策的依据。

- `AgentPhase`

  - `UNDERSTAND_PROJECT`
  - `DIAGNOSE`
  - `PLAN`
  - `EXECUTE`
  - `VERIFY`
  - `RECOVER`
  - `REPLAN`
  - `ASK_USER`
  - `SUMMARIZE`
  - `BLOCKED`
- `RuntimeStateMetadata`
  - `goal`
  - `phase`
  - `known_facts`
  - `unknowns`
  - `assumptions`
  - `resolved_questions`
  - `candidate_files`
  - `selected_files`
  - `planned_edits`
  - `modified_files`
  - `verification_status`
  - `tool_history`
  - `decision_history`
  - `risk_level`

- `ToolDecisionMetadata`

状态必须持续更新，不能只把工具输出附加到 prompt 或日志里。

## 4. 阶段图与动作类型

Runtime 使用阶段规划，而不是预先固定工具链。

默认阶段流：

```text
UNDERSTAND_TASK
  -> UNDERSTAND_PROJECT
  -> DIAGNOSE
  -> PLAN
  -> EXECUTE
  -> VERIFY
  -> SUMMARIZE
```

异常与分支：

- `RECOVER`：工具失败、验证失败、执行异常后进入。
- `REPLAN`：关键假设被推翻、风险升高、当前计划不可执行、目标已完成或验证失败不符合原计划时进入。
- `ASK_USER`：只有高影响、高风险或重大歧义需要用户决策时进入。
- `BLOCKED`：风险过高、预算耗尽、缺少必要权限、用户决策缺失且无法安全推进时进入。

每一步动作必须明确属于以下类型之一：

- `reason`：基于已有状态推理，不调用工具。
- `tool`：获取外部信息或执行非编辑型环境操作。
- `edit`：改变项目文件或项目状态。
- `verify`：验证外部状态或修改结果。
- `ask_user`：请求用户决策。
- `finish`：总结并结束。

Runtime 的控制器应优先选择 `reason`，只有当状态不足以支持下一步判断时才选择工具或询问用户。

## 5. Tool Router 与工具调用策略

模型不应直接自由决定工具调用。模型应表达 `DecisionNeed`，由 `ToolRouter` 选择工具。

`ToolRouter` 是 need 到工具选择的唯一入口。旧的直接 `tool_calls` 规划路径应删除；如果某些工具暂时无法表达为 `DecisionNeed`，应扩展 `DecisionNeedMetadata` 或 Router 规则，而不是继续保留旧工具调用协议。

推荐执行链：

```text
Model/Planner 提出 DecisionNeed
ToolRouter 映射为 ToolSelection
Guard 检查预算、风险、权限和用户确认需求
Executor 执行工具
StateUpdater 吸收结果
Controller 决定继续、验证、恢复、重规划或结束
```

Tool Router 必须支持：

- 信息缺口触发：项目结构、入口文件、配置方式、函数位置、文档约束等。
- 证据不足触发：准备下结论但证据不足时补充观察。
- 状态改变触发：写操作、依赖变更、配置变更后触发验证。
- 失败触发：工具、测试或命令失败后进入诊断，不盲目重试。
- 风险触发：核心架构、配置、依赖、权限、删除、批量重构等高风险区域触发更强证据或用户确认。

Tool Router 还应记录每次选择的解释：

- 为什么调用这个工具。
- 为什么现在调用。
- 为什么不用其他工具。
- 这次调用要解决什么不确定性。
- 工具结果会如何改变下一步计划。

## 6. 预算、成本与停止条件

工具调用必须有成本意识。成本包括时间、上下文污染、错误信息引入、执行风险、用户等待成本、项目状态改变风险和 token 消耗。

Runtime 必须实现预算：

- 最大工具调用次数。
- 最大文件读取数量。
- 最大编辑文件数量。
- 最大验证尝试次数。
- 最大恢复轮数。
- 最大重规划轮数。
- 最大连续无新事实工具调用次数。

停止条件：

- 已获得足够信息。
- 新增工具调用不再改变计划。
- 连续调用没有产生新事实。
- 候选范围已经足够小。
- 当前任务可以基于已有信息推进。
- 风险过高，需要停止或请求确认。
- 预算耗尽，需要总结当前状态。

Runtime 必须避免：

- 无限搜索更多上下文。
- 不停读取文件。
- 不停重试测试。
- 不停扩大修改范围。
- 没有新事实时反复重规划。

## 7. 状态吸收与观察优先级

工具结果不能只是放进上下文。每次工具调用后必须更新 runtime state。

任何旧的“把工具输出拼到 prompt/context 里但不更新状态”的逻辑都应删除或迁移到 `StateUpdater`。上下文构建只能消费 runtime state，不能成为事实来源。

工具结果应转化为：

- 新事实。
- 已解决的不确定性。
- 新增不确定性。
- 候选文件变化。
- 选中文件变化。
- 计划变化。
- 风险变化。
- 下一步动作变化。
- 验证状态变化。

观察优先级：

```text
工具观察结果
项目文件内容
测试/运行结果
文档说明
明确用户要求
模型推理
模型猜测
```

模型推理只能生成假设，不能替代证据。

## 8. 文件选择、编辑与职责边界

文件编辑是高风险动作，必须单独建模。

职责边界：

- `Planner`：决定当前阶段和子目标。
- `FileSelector`：选择候选文件并提供证据。
- `Guard`：判断是否允许编辑。
- `Editor`：只执行已批准的编辑计划。
- `Verifier`：验证结果。
- `Reporter`：总结证据、修改、验证和风险。

关键规则：

- 候选文件不等于待编辑文件。
- 文件选择必须有证据。
- Editor 不允许自己决定编辑哪些文件。
- 每轮编辑只处理一个清晰子目标。
- 默认使用最小编辑集。
- 高风险文件需要更强证据。

文件选择证据来源：

- 路径和命名。
- 文件内容。
- 函数或类定义。
- 调用链。
- 依赖关系。
- 测试失败信息。
- 文档说明。
- 历史修改记录。
- 配置引用。

编辑前必须形成 `EditPlanMetadata`：

- 为什么要改这个文件。
- 证据是什么。
- 打算改什么。
- 不允许改什么。
- 修改风险是什么。
- 如何验证。

任何旧 executor、task runner、improvement helper 如果自行决定编辑文件，应移除该职责。文件选择统一进入 `FileSelector`，编辑授权统一进入 `EditGuard`，实际写入工具只执行已批准计划。

## 9. 验证、失败诊断与恢复

只要发生以下动作，就必须进入验证：

- 编辑文件。
- 创建文件。
- 删除文件。
- 生成代码。
- 修改依赖。
- 修改配置。
- 执行可能改变项目状态的命令。

验证可按成本选择：

- 检查 diff。
- 读取修改后的文件确认一致性。
- 运行相关测试。
- 运行全量测试。
- 运行 lint。
- 运行示例命令。
- 静态检查。

但不能跳过验证。

验证失败后的流程：

1. 读取失败信息。
2. 定位失败类型。
3. 判断失败是否由刚才修改引起。
4. 找到最小相关文件。
5. 形成修复计划。
6. 执行最小修改。
7. 再次验证。

验证失败是新的信息源，不是继续随机编辑的许可。

旧验证路径如果只返回 pass/fail 而不产出失败类型、证据、相关文件和恢复入口，应删除或改造成 `RuntimeVerifier` / `StateUpdater` 的输入。

## 10. Replanning 策略

Replanning 是正常行为，但必须有边界。

触发条件：

- 关键假设被推翻。
- 发现任务风险高于预期。
- 当前计划无法执行。
- 工具结果显示目标已完成。
- 验证失败且原因不符合原计划。

约束：

- 不要每一步都重新规划。
- 不要因为小信息不断推翻整体方向。
- 没有新事实时不得重复重规划。
- Replanning 消耗预算。
- Replanning 后必须记录被替换假设、触发证据、新计划和风险变化。

## 11. 流式 Need 与中断机制

Runtime 应支持流式过程中发现信息缺口。

推荐模式：

```text
模型流式输出 need
系统暂停生成
ToolRouter 选择工具
Guard 审查
Executor 执行
StateUpdater 更新状态
模型继续生成
```

规则：

- 模型表达需要什么信息，而不是直接自由调用工具。
- 中断必须记录触发原因和恢复点。
- 中断也受预算、风险和权限控制。
- 工具结果必须更新 state 后才能继续生成。

## 12. 用户确认策略

用户确认只用于高价值、高风险、高影响决策，不能用于普通信息缺口。

应请求用户确认的情况：

- 多个方向价值接近，且选择显著影响项目方向。
- 需要删除或重写大量代码。
- 需要修改高风险配置。
- 需要引入新依赖或新框架。
- 需求存在重大歧义。
- 继续执行可能造成不可逆影响。

普通信息缺口优先通过工具解决。

## 13. 项目提升与创造性改进

对于“提升这个项目”“让项目更有用”“优化体验”“加点亮点”等任务，Runtime 不能直接从文件开始编辑。

必须先完成：

1. 理解项目目标。
2. 识别当前能力。
3. 分析用户价值。
4. 提出多个提升方向。
5. 评估价值、成本、风险。
6. 选择一个最小可实现方向。
7. 再进入文件选择和编辑。

创造性改进必须满足：

- 符合项目目标。
- 对用户有明确价值。
- 与目标 Runtime 架构一致。
- 能以较小步长实现。
- 可以验证。
- 不显著增加复杂度。

优先选择高价值、低到中等成本、风险可控、验证明确、与项目方向一致的改进。

## 14. 多轮闭环执行

复杂任务必须拆成多个闭环，而不是一次性完成。

标准闭环：

```text
理解 -> 计划 -> 执行 -> 验证 -> 总结
```

每个闭环只完成一个清晰子目标。

示例：

- 第一轮：理解项目并提出方向。
- 第二轮：实现一个小功能。
- 第三轮：验证并修复。
- 第四轮：补文档和测试。
- 第五轮：再次评估下一步。

每轮结束时必须记录：

- 本轮目标。
- 已收集证据。
- 修改文件。
- 验证结果。
- 残余风险。
- 下一轮建议。

## 15. 可解释报告

完成或停止时，Runtime 必须输出可解释报告。

报告内容：

- 原始目标。
- 最终阶段和停止原因。
- 已知事实和关键证据。
- 工具调用摘要及原因。
- 文件选择证据。
- 编辑计划和实际修改。
- 验证计划和验证结果。
- 失败、恢复和重规划记录。
- 用户确认记录。
- 剩余风险和建议下一步。

该报告用于用户审查，也用于调试 Agent Runtime 行为。

## 16. 目标工程组织

OpenPilot 应按以下职责组织；如果现有模块与这些职责重复，以本节目标结构为准进行删除或迁移：

- `AgentRuntimeController`
  - 维护阶段图、动作选择、停止条件、重规划和最终报告。
- `RuntimeStateMetadata`
  - 承载显式状态。
- `ToolRouter`
  - 将 `DecisionNeed` 映射到 `ToolSelection`。
- `RuntimeGuard`
  - 统一预算、权限、风险和用户确认策略。
- `FileSelector`
  - 从候选文件中选择可编辑文件并附证据。
- `EditGuard`
  - 批准或拒绝 `EditPlanMetadata`。
- `ToolEventLoopRunner`
  - 执行工具和状态更新闭环。
- `StateUpdater`
  - 吸收工具结果。
- `RuntimeVerifier`
  - 选择和执行验证计划。
- `RuntimeReporter`
  - 输出最终可解释报告。

这些职责可以拆成类、函数、状态机、middleware 或 policy layer，但每个职责必须只有一个权威归属。旧模块如果承担同一职责，应迁移能力后删除。

## 17. 分阶段实施计划

### Phase 1：状态合约与运行时骨架

- 新增 runtime metadata。
- 新增 phase graph。
- 新增 runtime budget。
- 新增 decision history 和 tool history。
- `IntelligentAutopilot.execute()` 不再承担流程控制，只作为薄入口调用 `AgentRuntimeController.run()`；如果其内部 helper 与 Runtime Controller 重复，应迁移后删除。

### Phase 2：Need 驱动工具选择

- 修改 tool planning prompt：只输出 `decision_needs`。
- `ToolRouter.route(state, need) -> list[ToolSelection]` 成为 need 到工具的唯一入口。
- 每个工具选择必须记录解释、成本、预算和风险。
- 删除模型直接自由选择工具的旧 `tool_calls` 协议、解析逻辑和测试。

### Phase 3：状态吸收与停止条件

- 每次工具执行后调用 `StateUpdater`。
- 将工具结果转化为事实、问题、候选文件、计划变化和风险变化。
- 实现连续无新事实、预算耗尽、风险过高等停止条件。
- 工具失败进入诊断，不盲目重试。

### Phase 4：证据驱动文件选择与受控编辑

- 引入 `FileSelector`。
- `candidate_files` 必须通过证据升级为 `selected_files` 后才可编辑。
- `EditPlanMetadata` 成为编辑前置条件。
- `EditGuard.approve(state, edit_plan) -> GuardDecisionMetadata` 控制所有写操作和项目状态变更命令。
- Editor 只执行已批准计划。
- 删除任何直接从 task description、文件名猜测或 executor 内部自行决定编辑文件的旧路径。

### Phase 5：验证与恢复闭环

- 所有写操作后强制进入 `VERIFY`。
- `RuntimeVerifier` 根据成本选择 diff、相关测试、全量测试、lint、示例命令、静态检查或读取确认。
- 验证失败进入 `RECOVER`，先诊断再修复。
- 修复也必须走最小编辑集和再次验证。

### Phase 6：Replanning、流式中断与用户确认

- 实现有限 Replanning。
- 支持流式 `need -> router -> tool -> state update -> continue`。
- 引入高风险用户确认策略。
- 普通信息缺口不问用户，优先工具解决。

### Phase 7：项目提升任务策略

- 对提升类任务先确定提升方向，再选文件。
- 评估价值、成本、风险、验证方式和目标架构一致性。
- 每轮只实现一个最小可验证改进。

### Phase 8：可解释报告与审计

- 输出工具调用解释、文件选择证据、编辑计划、验证结果、失败诊断、重规划记录和残余风险。
- README 更新实际架构。
- 删除旧报告路径；最终结果统一由 `RuntimeReporter` 从 runtime state 生成。

### Phase 9：旧结构删除与收敛

- 删除与 `AgentRuntimeController` 职责重复的旧 runner/helper。
- 删除与 `ToolRouter` 职责重复的工具选择 prompt、解析器和 fallback 逻辑。
- 删除与 `StateUpdater` 职责重复的上下文拼接式状态管理。
- 删除与 `EditGuard` / `RuntimeGuard` 职责重复的局部权限、风险和预算判断。
- 删除与 `RuntimeVerifier` 职责重复的验证入口。
- 删除只为旧路径服务的测试和文档，并用新 Runtime 行为测试替代。

## 18. Public Interfaces

核心接口：

- `AgentRuntimeController.run(goal, context) -> dict`
- `ToolRouter.route(state, need) -> list[ToolSelection]`
- `StateUpdater.apply_tool_result(state, selection, result) -> RuntimeStateMetadata`
- `EditGuard.approve(state, edit_plan) -> GuardDecisionMetadata`
- `RuntimeVerifier.plan(state, context) -> VerificationPlanMetadata`
- `RuntimeReporter.report(state) -> RuntimeReportMetadata`

破坏性调整要求：

- `IntelligentAutopilot.execute()` 可以改变返回结构；新结构必须包含 runtime state、工具解释、编辑计划、验证结果和风险报告。
- CLI 命令名、参数和输出格式都可以按新 Runtime 重新设计；不为了旧 CLI 语义保留内部兼容路径。
- 删除旧 `tool_calls` 公共输入；外部调用方必须迁移到 `decision_needs` 或更高层 goal 接口。
- 不保留仅用于兼容旧结构的 adapter、alias 或 shim。

## 19. Test Plan

必须覆盖以下测试：

- Runtime metadata 序列化、默认值、预算消耗和停止条件。
- Phase graph 正常推进、失败进入 `RECOVER`、风险或预算进入 `BLOCKED`。
- `DecisionNeed` 被 `ToolRouter` 映射为正确工具。
- 旧 `tool_calls` 输入被拒绝或不存在解析入口。
- Router 在预算耗尽、高风险、权限不足时拒绝或要求确认。
- 工具选择解释被记录。
- 工具结果被吸收为事实、已解决问题、新问题、候选文件、风险变化和下一步动作。
- `candidate_files` 不能直接编辑，必须升级为 `selected_files`。
- 无证据、无范围、无验证计划、超过最小编辑集的编辑被拒绝。
- Editor 无权选择目标文件。
- 写文件、删文件、依赖变更、配置变更、代码生成、变更型命令后必须验证。
- 验证失败先诊断，再最小修复，再次验证。
- Replanning 只在触发条件满足时发生，并消耗预算。
- 流式 need 中断能恢复生成，并更新状态。
- 用户确认只在高风险或高影响决策触发。
- 项目提升任务先形成提升方向，再选择文件。
- 创造性改进必须符合目标、价值、架构、风险和验证约束。
- 复杂任务拆成多个闭环，每个闭环可验证。
- 最终报告包含证据、工具调用原因、修改、验证和风险。
- 验证重复旧结构已删除：不存在第二套工具规划、第二套编辑授权、第二套验证入口或第二套报告汇总。
- 回归新 `ToolEventLoopRunner`、`AgentRuntimeController`、CLI、metadata contract、project evaluator 和 autonomous iteration 测试。

## 21. Assumptions

- 不引入新第三方依赖，优先复用 Pydantic v2、现有 tool metadata 和现有执行器。
- 可以删除 `autonomous_iteration` 中与目标 Runtime 作用相同的旧结构；不因为历史路径存在就保留。
- 删除旧 `tool_calls` 协议，以 `decision_needs` 作为模型到工具选择的唯一协议。
- Guard 采用确定性规则；高风险确认和用户交互必须归入统一 RuntimeGuard，不允许散落在旧执行器里。
- 流式中断、报告、验证和恢复如果已有旧实现，必须迁移到目标职责组件后删除旧入口。
- 测试以目标架构行为为准；旧兼容行为不再作为通过条件。
