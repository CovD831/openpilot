\section{项目概述}

\subsection{项目名称}
\textbf{OpenPilot Personal Agent}（面向垂类知识工作的可靠工作流生成与管理平台）。

\subsection{项目定位}

\textbf{OpenPilot 不是一个单点研究报告工具，而是面向垂类知识工作的可靠工作流生成与管理平台}。用户通过目标描述、资料投喂、交互澄清、反馈修正和权限授权，引导系统生成一个可复用、可审计、能持续积累上下文的个人智能工作流；研究报告只是第一个可验证模板，用来证明这套工作流生成与执行闭环可以稳定落地。

本文中的 Agents 本质上是增强后的传统工作流：它不仅包含固定步骤，还包含 LLM 推理、上下文记忆、工具调用、权限控制、日志复盘和用户反馈迭代。面向非技术读者时，OpenPilot 统一将其称为“智能工作流”或“增强型工作流”；在工程实现层，仍保留 Agent 作为内部组件名。

在小众专业领域里，通用大模型常常缺少足够的世界知识、技能知识和项目长期上下文。OpenPilot 通过资料阅读、工具执行、记忆沉淀、日志复盘和授权范围内的自主探索，让智能工作流逐步补齐某一垂类任务所需的知识、流程和操作能力。这不是简单的提示词工程，而是把用户经验、领域资料和工具调用组织成可持续改进的工作流系统。

\textbf{核心价值}：让个人和小团队能够在引导与交互下生成可靠工作流，把“我知道怎么做但每次都要从头教 AI”的重复过程，沉淀成可追踪、可复用、可复盘的专业工作流；再进一步扩展到主动在线助理、后台自动整理、超级上下文管理和垂类专业任务执行。

OpenPilot 的核心闭环可以概括为“用户目标、工作流生成、工具执行、日志复盘、记忆更新、后台整理”的持续循环，如图~\ref{fig:core-loop} 所示。

\textbf{阶段策略}：
\begin{enumerate}
  \item \textbf{Phase 1: 引导式可靠工作流生成} —— 从研究、文档、资料整理等场景切入，让用户通过交互把任务流程固化为可复用智能工作流（当前 MVP）
  \item \textbf{Phase 2: 主动在线助理与超级上下文管理} —— 在授权边界内后台整理资料、跟踪项目状态、维护长期记忆，并主动提出下一步建议（核心目标）
  \item \textbf{Phase 3: 小众垂类专业工作流执行} —— 面向科研、投研、法律、医疗资料分析、工程开发和企业知识库等领域，支持工作流在授权资料和用户反馈下补齐知识与技能（未来扩展）
\end{enumerate}

\textbf{长期愿景与终极目标}：OpenPilot 的终极目标不是做一个更会聊天的 AI，而是探索一条从可靠智能工作流走向个人级通用智能助理的路径。AGI 在本文中被视为长期方向，而不是当前 MVP 或本轮融资的交付承诺：系统需要在可控边界内持续积累世界知识、技能知识、用户偏好、任务经验和工具能力，逐步提升跨领域理解、规划、执行和自我修正能力。

早期融资材料的重点不是宣称“什么都能做”，而是证明用户可以通过引导和反馈生成一个稳定工作的智能工作流，并在真实任务中持续积累可迁移的记忆、工具协议、执行日志和领域能力。

\subsection{一句话简介}

OpenPilot 是面向垂类知识工作的可靠工作流生成与管理平台：用户通过目标描述、资料投喂和交互反馈生成专属智能工作流，系统负责超级上下文管理、工具执行、日志复盘和后台整理，让工作流在授权边界内持续补齐领域知识与任务技能。

\subsection{融资阶段定位}

项目当前适合作为种子轮或天使轮早期项目推进，重点验证三件事：
\begin{enumerate}
  \item 用户能否通过引导式交互，把一个垂类任务流程生成可复用、可审计的智能工作流；
  \item 智能工作流能否在授权资料和工具边界内完成研究报告、资料整理、文档自动化等首批模板任务；
  \item 超级上下文、记忆与复盘机制能否显著提升下一次相似任务的执行质量。
\end{enumerate}

\section{市场机会}

\subsection{行业现状}

2024--2026 年，AI 个人助理、Agent 和智能工作流产品正在从“对话问答”向“任务执行”演进。现有市场大致分为四类：

\begin{enumerate}
  \item \textbf{通用 Chatbot}：如 ChatGPT、Claude、Gemini 等，覆盖面广，但多数任务仍需要用户逐轮指挥，长期任务记忆与跨工具执行能力仍有限。
  \item \textbf{编码 Agent 与开发者工具}：如 OpenCode、Claude Code、OpenAI Codex/GPT、GitHub Copilot、Cursor、Windsurf 等，在编程场景表现出色，但能力边界集中在终端、IDE、代码仓库与软件工程上下文。
  \item \textbf{Agent 开发框架}：如 LangChain、CrewAI、AutoGPT 类框架，适合开发者搭建自动化流程，但普通用户配置和维护成本较高。
  \item \textbf{桌面级 Copilot}：依托操作系统或办公套件集成，体验自然，但生态封闭，用户难以自由接入自己的模型、脚本、企业工具和私有数据。
\end{enumerate}

OpenPilot Personal Agent 面向这四类产品之间的空白地带：它不是单纯聊天工具，也不是只给开发者使用的框架，而是帮助用户在具体垂类任务中生成可靠、长期可用、可审计、可迁移的个人智能工作流。研究报告、资料整理和文档自动化是最早验证的模板，而不是项目能力边界。

\subsection{竞品对比}

OpenClaw 以及以 OpenCode、Claude Code、OpenAI Codex/GPT 系列开发工具为代表的编码 Agent，体现了当前 Agent 产品的两个重要方向。OpenClaw 的优势在于本地运行、聊天渠道接入和真实个人助理动作，例如通过 Telegram、WhatsApp 等入口处理邮件、日历和个人工作流；但其现有形态也暴露出若干不足：一是入口偏聊天化，适合触发简单动作，却不天然适合承载复杂研究、资料整理和多阶段任务管理；二是执行过程如果缺少结构化规划、权限分级、任务日志和复盘机制，用户很难判断 Agent 为什么这么做、做到了哪一步、失败后如何恢复；三是个人知识、长期偏好、历史任务经验与工具调用之间的记忆闭环仍需进一步产品化，否则系统容易停留在“会操作工具”的助理，而不是“越用越懂用户”的个人执行代理。编码 Agent 则聚焦终端、IDE、桌面开发环境和代码仓库自动化。

因此，本项目的机会不在于简单复制 OpenClaw 的聊天入口，也不在于和 OpenCode、Claude Code、OpenAI Codex/GPT 等编码 Agent 在软件工程场景正面硬碰，而在于补齐现有产品中“用户无法稳定沉淀自己的智能工作流、复杂任务承载不足、执行过程不可复盘、长期记忆弱、权限边界不够显性”的短板。OpenPilot 的 MVP 明确收敛为引导式可靠工作流生成：先把“用户目标 → 交互澄清 → 工作流生成 → 工具执行 → 日志复盘 → 记忆更新 → 后台持续整理”做成稳定闭环；研究报告只是首个低风险、可检查的模板。主要竞品与差异化定位见表~\ref{tab:competitors}。

\subsection{核心竞争力}

\subsubsection{智能增强框架，而非简单工具}

\textbf{传统 AI 助手的问题}：
\begin{itemize}
  \item 依赖模型本身的能力上限
  \item 难以沉淀用户反馈和任务经验
  \item 缺乏系统化的任务拆解
  \item 缺少可审计的执行边界
\end{itemize}

\textbf{OpenPilot 的解决方案}：

\begin{center}
\fbox{\parbox{0.9\textwidth}{
\centering
普通 LLM 模型 + OpenPilot 框架 = 可记忆、可执行、可审计的知识工作流代理
}}
\end{center}

\textbf{关键优势}：
\begin{itemize}
  \item \textbf{模型无关}：支持任何 OpenAI-compatible LLM
  \item \textbf{能力增强}：通过上下文、记忆和工具协议提升普通模型在真实任务中的稳定性
  \item \textbf{复盘改进}：从执行日志和用户反馈中提取可复用经验
  \item \textbf{安全可控}：完善的权限系统和风险控制
\end{itemize}

\subsection{Phase 2: 主动在线助理与超级上下文管理（核心目标）}

\textbf{定位}：把 LLM 调用组织成稳定、可审计、可回放的在线助理管线，让智能工作流能够基于超级上下文理解项目状态，在授权边界内后台自动整理资料、提出下一步建议，并执行低风险子任务。

\textbf{核心能力}：

\begin{enumerate}
  \item \textbf{Context Loader}：加载固定 system prompt、压缩对话历史、相关文件、相关记忆、虚拟环境信息和当前项目状态
  \item \textbf{Goal Maker}：基于上下文生成项目整理方向、待补知识、待确认问题和下一步建议
  \item \textbf{Task Designer}：将建议转化为具体、可授权、可回放的任务
  \item \textbf{Task Decomposer}：结合难度评估，把任务递归拆解为更容易执行的子任务
  \item \textbf{Task Executor}：调用 File Reader、File Writer、Command Executor、Embedder 等工具完成子任务
  \item \textbf{后台整理与记忆更新}：记录异构日志，并把有价值的执行经验、资料摘要、状态变化和用户反馈写入记忆与项目上下文
\end{enumerate}

\textbf{能力对比示例}：

\begin{quote}
\textbf{用户}：”帮我建立一个新能源汽车出海研究工作流，以后持续跟踪政策、竞品和市场机会”

\textbf{传统 AI}：”好的，我可以帮你...”

\textbf{OpenPilot 主动在线助理管线}：
\begin{enumerate}
  \item Context Loader 载入历史研究、用户偏好、授权文件、记忆和项目状态
  \item Goal Maker 生成政策跟踪、竞争格局、市场规模、风险案例和资料缺口等方向
  \item Task Designer 把方向转化为资料检索、文件阅读、引用整理、报告撰写和后台更新任务
  \item Task Decomposer 递归拆解复杂研究任务，直到子任务足够容易执行
  \item Task Difficulty Evaluator 判断是否需要继续拆解
  \item Task Executor 调用文件、命令和语义检索工具完成资料读取与整理
  \item Memory Module 记录领域术语、可信来源、用户判断标准、失败原因和可复用流程
  \item 系统输出带引用来源的阶段性报告、执行日志、复盘结果和下一步建议
\end{enumerate}
\end{quote}

\textbf{商业价值}：
\begin{itemize}
  \item 自动化 50-70\% 的重复性任务
  \item 减少人工确认次数 70\%
  \item 提升任务完成速度 3-5 倍

\subsection{工程架构}

OpenPilot 的技术设计以 \texttt{instructions/openpilot} 为最高优先级来源。系统不是把所有能力堆进一个单体 Agent，而是把项目拆成模块、Agent、公共工具和实用函数四类边界，并要求每个模块、Agent、工具和函数提供明确日志，便于调试、复盘和后续扩展。

为避免概念混淆，本文在产品层使用“增强型工作流”描述用户可感知的能力；在工程层保留 \texttt{Agent} 作为实现概念。一个增强型工作流可以由一个或多个 Agent、Tools、Memory 和 Utils 共同组成，Agent 负责承载 LLM prompt、函数调用和工具编排。

\begin{verbatim}
User Goal
   |
Context Loader
   |
Goal Maker
   |
Task Designer
   |
Task Decomposer
   |
Task Executor
   |-- File Reader
   |-- File Writer
   |-- Command Executor
   `-- Embedder
   |
Memory / Project / Environment Update
   |
Final Output
\end{verbatim}

\subsection{核心模块设计}

\subsubsection{Memory Module}

Memory Module 是 OpenPilot 的核心能力之一，负责保存对话上下文、用户偏好、项目结构和虚拟环境信息。它不是单纯的聊天记录缓存，而是后续任务拆解、文件定位、工具选择和偏好判断的上下文来源。

\textbf{Context Manager} 负责存储和管理上下文。系统对用户话语和 Agent 话语使用不同策略：用户话语原样保留，确保用户指令不被压缩改写；Agent 历史通过压缩函数整理，以节省上下文窗口。输出上下文可以表示为：

\[
Context=[UH_1,f_{Comp}(AH_1),UH_2,f_{Comp}(AH_2), ...].
\]

其中 \texttt{Agent History Compressor} 的输入是 Agent History，输出是 Compressed Agent History；\texttt{Output Context} 返回当前已存储和拼接的上下文。

\textbf{Memory Vault} 类似人的长期记忆，用于保存用户偏好和值得长期记住的知识。当系统对用户偏好缺乏把握时，Memory Vault 可以发起询问，并在需要时把答案写入 memory graph。Memory graph 的节点、链接和函数边界见表~\ref{tab:memory-vault-graph}。

\begin{table}[htbp]
  \centering
  \small
  \begin{tabularx}{\textwidth}{p{3.4cm} X}
    \toprule
    \textbf{Memory Graph 元素} & \textbf{属性与含义} \\
    \midrule
    Memory Node & \texttt{content} 保存声明式记忆内容；\texttt{semantic\_info} 保存语义向量；\texttt{remind\_frequency} 记录该记忆被提醒的频率；\texttt{last\_updated\_timestamp} 记录最近更新时间。 \\

\textbf{Project Manager} 负责管理项目上下文，把项目文件信息提取到每级目录的 \texttt{sketch.json} 中，并在需要时检索相关文件。每个 \texttt{sketch.json} 维护的核心结构是：

\begin{center}
\texttt{filename}: $EV_{func}$
\end{center}

其中 \texttt{filename} 是标准文件名，$EV_{func}$ 是该文件功能描述的 embedding vector。文件变更时，\texttt{Update Function} 根据变更路径定位对应目录并更新 \texttt{sketch.json}；查询到来时，\texttt{Search Function} 计算 query embedding 与文件功能向量的距离，返回相关 file paths。

\textbf{Virtual Environment Manager} 负责把项目虚拟环境纳入记忆范围。它提供 \texttt{Environment Creator} 创建当前项目虚拟环境，\texttt{Packet Installer} 根据 packet names 输出 bash install commands，\texttt{Get Environment Context} 返回当前环境包信息。

\subsubsection{Autonomous Iteration Module}

Autonomous Iteration Module 是实现受控迭代执行的核心模块，用稳定管线为项目生成改进方向、任务和可执行子任务。用户发起 query 后，系统按以下流程运行；各 Agent / Function 的职责见表~\ref{tab:autonomous-iteration}：

\begin{verbatim}
Run Context Loader;
Run Goal Maker;
for each goal:
  Run Task Designer;
  for each task:
    Run Task Decomposer until the subtask is easy to be solved;
    for each subtask:
      Run Task Executor;
\end{verbatim}

\begin{table}[htbp]
  \centering
  \small
  \begin{tabularx}{\textwidth}{p{3.4cm} X}
    \toprule
    \textbf{Agent / Function} & \textbf{职责} \\
    \midrule

    Goal Maker & 基于上下文和固定 prompt 生成项目改进方向，例如改善 UI、添加功能或清理代码。 \\

    Task Difficulty Evaluator & 输入 context 和 task，输出 difficulty，用于判断是否继续拆解。 \\
    Task Executor & 接收 context 和 subtask，并调用合适工具完成子任务。 \\

\subsection{核心差异化}

\begin{enumerate}
  \item \textbf{不是单点工具，是模块化框架} —— 用清晰的 Module、Agent、Tool 和 Utility 边界组织能力
  \item \textbf{不是一次性回答，是任务执行闭环} —— 通过上下文加载、目标生成、任务设计、递归拆解和执行推进任务
  \item \textbf{不是孤立对话，是可复用记忆} —— 通过 Memory Vault、Project Manager 和虚拟环境上下文积累经验
  \item \textbf{不是黑盒执行，是透明日志} —— 每个模块、Agent、工具和函数都应具备可调试日志
  \item \textbf{不是无边界自动化，是可控授权} —— 通过风险分级和确认机制约束真实操作
\end{enumerate}

\section{风险与应对}

\subsection{主要风险}

\begin{enumerate}
  \item \textbf{安全风险}：代理具备执行能力后，错误操作可能导致文件丢失、误发邮件或账号风险。
  \item \textbf{可交付性风险}：如果早期追求过宽泛的代理能力，容易陷入范围过大、质量不可控的问题。
  \item \textbf{模型不稳定风险}：LLM 可能幻觉、误判工具结果或生成不可执行计划。
  \item \textbf{隐私风险}：长期记忆涉及用户工作内容、偏好和敏感信息，需要严格存储与授权机制。
  \item \textbf{市场教育风险}：用户可能不愿立即授权 AI 操作电脑，需要通过透明日志和低风险场景建立信任。
\end{enumerate}

\subsection{改良策略}

\begin{itemize}
  \item MVP 聚焦引导式可靠工作流生成和首批低风险模板，不承诺无边界个人助理或全场景跨工具自动化；
  \item 所有关键操作保留日志，用户可查看计划、工具调用、执行结果和记忆更新；
  \item 低风险任务按授权规则执行，中风险任务通知，高风险任务确认，危险任务禁止；
  \item 支持沙盒执行、撤销或回滚机制，减少真实环境误操作；
  \item 记忆写入需要可解释，用户可查看、编辑、删除和导出记忆；
  \item 即使以 AGI 作为长期终极目标，当前产品仍坚持授权边界、审计日志、人工确认和可回滚机制，不构建不受控的自主执行系统；
  \item 对市场数据和商业预测进行来源核验，融资材料中避免无依据承诺。
\end{itemize}

\section{结论}

OpenPilot Personal Agent 的机会不在于宣称“替用户做任何事情”，而在于让用户能够在引导与交互下生成可靠、可审计、能长期积累上下文的个人智能工作流。它先用研究报告、资料整理和文档自动化等低风险模板验证“用户目标、工作流生成、工具执行、日志复盘、记忆更新、后台整理”的闭环；当这个闭环形成稳定留存和付费后，再扩展到主动在线助理、超级上下文管理和小众垂类专业工作流执行。

因此，OpenPilot 当前的商业化抓手是可靠工作流生成与管理，长期愿景则是让这些可控、可复用、可复盘的工作流逐步成为个人级 AGI 的能力底座。
