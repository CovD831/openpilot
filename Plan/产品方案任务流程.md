# OpenPilot 产品方案任务流程（Phase 2: AGI 智能体能力）

## 项目定位

**OpenPilot 不是简单的 AI 助手，而是智能增强框架**：

```
普通 LLM 模型 + OpenPilot 框架 = 接近 AGI 的超级智能体
```

本文档是 OpenPilot 的详细实现指南。**Phase 1 基础层已完成**，当前聚焦 **Phase 2: AGI 智能体能力**，目标是实现自主思考、工具编排、代码执行和持续学习，让普通模型获得接近 AGI 的问题解决能力。

**核心差异化**：
- **不是 AI 壳子**：不是在单一场景（会议、邮件、日程）内提供 AI 能力，而是跨场景的智能体框架
- **不是功能增强**：不是在现有工具上叠加 AI 功能，而是让普通模型获得接近 AGI 的能力
- **不是静态工具**：不是功能相对固定的产品，而是从每次执行中学习、持续优化的进化系统
- **不是数据孤岛**：不是各个工具的数据相互隔离，而是四层记忆系统跨场景共享经验
- **不是被动响应**：不是等待用户触发预定义功能，而是自主拆解任务、编排工具、执行计划

**关于整体架构**：关于 OpenPilot 的三阶段统一架构、核心竞争力和长期路线图，请参考 `OpenPilot-产品架构与路线图.md` 和 `项目说明书.tex`。

## OpenPilot 三阶段架构

1. **Phase 1: 个人助手**（✅ 已完成）
   - 定位：任务管理、进度跟踪、日报周报、记忆系统基础
   - 目标：构建智能增强框架的基础层，验证核心机制
   - 成果：四层记忆系统、置信度驱动自主度、任务推进核心流程
   - 商业价值：提升个人效率 30-50%

2. **Phase 2: AGI 智能体**（🚀 当前阶段）
   - 定位：自主思考、任务拆解、工具编排、代码执行、持续学习
   - 目标：让普通模型获得接近 AGI 的问题解决能力
   - 核心能力：工具自动执行、代码生成与运行、复杂问题分解、反思与优化
   - 商业价值：让 GPT-3.5 级别模型达到接近 GPT-4 的效果

3. **Phase 3: GUI 界面**（未来扩展）
   - 定位：可视化交互、补足命令行不足、降低使用门槛
   - 目标：让非技术用户也能使用智能增强框架
   - 商业价值：扩大用户群体，提升商业化潜力

## Phase 2 核心目标

Phase 2 的目标是**让普通模型获得接近 AGI 的问题解决能力**，需要实现以下核心能力：

### 1. 工具编排与自动执行
- **工具注册与发现**：插件式工具注册，支持动态扩展
- **智能工具选择**：基于任务需求自动匹配最佳工具组合
- **安全执行策略**：基于风险等级和置信度的分级执行策略
- **执行结果验证**：自动检查执行结果，识别失败并重试

### 2. 代码生成与执行
- **代码生成能力**：根据任务需求生成 Python/Shell 脚本
- **沙箱执行环境**：安全隔离的代码执行环境
- **执行结果分析**：解析执行输出，提取关键信息
- **错误处理与修复**：识别错误原因，自动修正代码

### 3. 复杂问题分解
- **多步骤规划**：将复杂问题分解为可执行的步骤序列
- **依赖关系管理**：识别步骤间的依赖，优化执行顺序
- **并行执行优化**：识别可并行执行的步骤，提升效率
- **动态调整能力**：根据执行结果动态调整后续计划

### 4. 反思与持续优化
- **执行结果评估**：自动评估任务完成质量
- **失败原因分析**：深度分析失败原因，提取经验教训
- **策略优化建议**：基于历史数据提出改进建议
- **记忆系统更新**：将经验写入记忆，提升后续任务表现

**Phase 2 的战略意义**：
- 从”任务管理助手”升级为”问题解决智能体”
- 从”被动响应”升级为”主动执行”
- 从”单次任务”升级为”持续学习”
- 证明”让普通模型变强大”的核心价值

## 1. Phase 2 智能体主流程

用户目标进入系统后的 AGI 智能体执行流程（体现问题解决能力）：

### 1.1 任务理解与记忆检索（继承 Phase 1）
1. 接收用户的高层项目或任务目标
2. **从四层记忆系统检索相关经验**：
   - 短期记忆：当前会话上下文
   - 长期记忆：用户偏好、常用约束
   - 任务记忆：相似任务的历史执行
   - 技能记忆：可复用的流程模板
3. 通过对话澄清截止时间、范围、优先级、依赖关系、可用资源和风险边界
4. 目标理解模块生成结构化任务卡

### 1.2 智能规划与工具编排（Phase 2 增强）
5. 时间线规划模块生成任务树、里程碑、依赖关系和执行计划
6. **工具编排模块**：
   - 为每个步骤匹配最佳工具组合
   - 识别可并行执行的步骤
   - 生成工具调用序列和参数
7. **自主判断模块计算置信度**：
   - 基于历史成功率、用户偏好匹配度、风险等级
   - 决定每个步骤的自主度：自动执行、通知后执行、确认后执行、人工介入
8. **代码生成模块**（新增）：
   - 识别需要代码解决的步骤
   - 生成 Python/Shell 脚本
   - 预估执行风险和资源需求

### 1.3 自动执行与验证（Phase 2 核心）
9. **执行器根据置信度和权限策略自动执行**：
   - 低风险高置信度：直接执行
   - 中风险高置信度：通知后执行
   - 高风险：确认后执行
   - 禁止操作：阻断
10. **代码执行引擎**（新增）：
    - 在沙箱环境中运行生成的代码
    - 捕获执行输出和错误
    - 超时和资源限制保护
11. **结果验证模块**（新增）：
    - 检查执行结果是否符合预期
    - 识别失败原因
    - 决定是否重试或调整策略
12. 任务日志模块记录计划、执行、结果、失败原因和用户反馈

### 1.4 反思与持续优化（Phase 2 增强）
13. **反思模块深度分析**：
    - 什么有效、什么失败、为什么失败
    - 工具选择是否最优
    - 代码质量和执行效率
    - 用户偏好变化
14. **策略优化模块**（新增）：
    - 生成改进建议
    - 更新工具选择策略
    - 优化代码生成模板
    - 调整置信度阈值
15. **记忆系统更新**：
    - 保存成功经验到技能记忆
    - 更新用户偏好
    - 记录失败教训
    - 提升后续任务置信度

### 1.5 持续进化
16. 每次执行后，系统学习并优化：
    - 工具使用效率提升
    - 代码生成质量提升
    - 问题分解能力提升
    - 自主度渐进提升
17. CLI 和 JSONL 日志展示完整执行过程和学习轨迹

**Phase 2 核心差异**：
- 不只是规划，而是**真正执行**
- 不只是选择工具，而是**编排工具链**
- 不只是记录经验，而是**主动优化策略**
- 不只是低风险自动，而是**中风险也能自动**（高置信度）

## 2. Phase 2 实现任务队列

本节按优先级排序，聚焦 Phase 2 的问题解决能力提升。

### 第一优先级：工具执行链路（核心能力）

#### OP-20 工具注册表增强

目标：建立完整的工具注册、发现和管理机制，支持动态扩展。

输入：
- 工具定义（名称、描述、版本、权限等级、输入输出 schema）
- 工具实现（可执行的函数或脚本）
- 工具元数据（失败模式、降级策略、审计要求）

输出：
- `ToolDefinition`：完整的工具定义
- `ToolRegistry`：工具注册中心
- `ToolCapability`：工具能力描述（用于智能匹配）

建议实现：
- 基于 OP-05 的基础，增强为可执行的工具注册表
- 支持工具插件动态加载
- 工具分类：`llm_tools`（LLM 调用）、`local_tools`（本地文件/脚本）、`web_tools`（网络请求）、`code_tools`（代码执行）
- 内置工具：`llm_summarizer`、`file_reader`、`file_writer`、`web_search`、`python_executor`、`shell_executor`
- 每个工具必须声明：权限等级、超时时间、资源限制、失败重试策略

验收标准：
- 可以注册新工具并查询工具列表
- 工具按能力标签分类（文件操作、网络请求、代码执行等）
- 高风险工具有明确的权限标记
- 工具可以声明依赖关系（例如 `web_search` 依赖网络访问）

测试：
- 注册内置工具
- 查询工具能力
- 权限等级验证
- 工具依赖检查

#### OP-21 智能工具选择与编排

目标：基于任务需求自动选择最佳工具组合，生成工具调用序列。

输入：
- `ExecutionPlan`（任务执行计划）
- `ToolRegistry`（可用工具列表）
- `MemoryQueryResult`（历史工具使用经验）
- 任务类型、资源需求、约束条件

输出：
- `ToolOrchestrationPlan`：工具调用序列
- `ToolSelection`：每个步骤的工具选择（包含备选方案）
- `ParallelExecutionGroup`：可并行执行的工具组

建议实现：
- 基于 OP-06 的基础，增强为智能编排
- 不只是单步匹配，而是生成完整的工具调用链
- 识别可并行执行的步骤（例如多个独立的文件读取）
- 为每个工具选择生成备选方案（主工具失败时的降级策略）
- 从任务记忆中学习最优工具组合
- 考虑工具执行成本（时间、资源、API 调用费用）

验收标准：
- 研究任务能生成：`web_search` → `llm_summarizer` → `file_writer` 的工具链
- 数据分析任务能生成：`file_reader` → `python_executor` → `llm_summarizer` 的工具链
- 识别可并行的文件读取操作
- 为高风险工具提供降级方案（例如 `file_writer` 失败时提示用户手动保存）

测试：
- 不同任务类型的工具选择
- 并行执行识别
- 备选方案生成
- 历史经验复用

#### OP-22 安全执行器

目标：在受控环境中安全执行工具，支持超时、资源限制、错误处理。

输入：
- `ToolOrchestrationPlan`（工具编排计划）
- `AutonomyDecision`（自主度决策）
- 用户确认状态

输出：
- `ExecutionResult`：执行结果（成功/失败、输出、错误信息）
- `ExecutionLog`：详细执行日志
- `ResourceUsage`：资源使用情况（时间、内存、API 调用）

建议实现：
- 基于 OP-07 的基础，增强为真正可执行
- 执行前检查：权限验证、资源可用性、依赖满足
- 执行中监控：超时控制、资源限制、异常捕获
- 执行后验证：结果检查、副作用确认、日志记录
- 支持并行执行（使用线程池或异步）
- 失败重试机制（指数退避）
- 沙箱隔离（文件系统、网络访问限制）

验收标准：
- 低风险工具（文件读取、LLM 调用）可以自动执行
- 中风险工具（文件写入）在高置信度时可以通知后执行
- 高风险工具（删除、发送消息）必须确认后执行
- 执行超时会被正确中断
- 执行失败会记录详细错误信息并尝试降级方案
- 所有执行都有完整审计日志

测试：
- 成功执行路径
- 超时中断
- 权限阻断
- 失败重试
- 并行执行
- 资源限制

#### OP-23 代码生成与执行引擎

目标：根据任务需求生成 Python/Shell 代码，在沙箱中安全执行。

输入：
- 任务描述
- 输入数据
- 执行约束（超时、资源限制、允许的库）

输出：
- `GeneratedCode`：生成的代码
- `CodeExecutionResult`：执行结果
- `CodeQualityReport`：代码质量评估

建议实现：
- 新增 `code_generator.py` 和 `code_executor.py`
- 代码生成：使用 LLM 生成 Python/Shell 脚本
- 代码审查：静态分析，检查危险操作（`os.system`、`eval`、文件删除等）
- 沙箱执行：使用 `subprocess` 或 Docker 容器隔离执行
- 输出捕获：捕获 stdout、stderr、返回值
- 错误处理：解析错误信息，生成修复建议
- 代码缓存：保存成功的代码到技能记忆

验收标准：
- 可以生成简单的数据处理脚本（读取 CSV、计算统计、生成图表）
- 可以生成文件操作脚本（批量重命名、格式转换）
- 危险操作（删除、网络请求）会被阻断或需要确认
- 执行超时会被正确中断
- 执行错误会生成可读的错误报告
- 成功的代码会保存到技能记忆供后续复用

测试：
- 代码生成质量
- 危险操作检测
- 沙箱隔离
- 超时控制
- 错误处理
- 代码复用

### 第二优先级：结果验证与优化

#### OP-24 执行结果验证模块

目标：自动检查执行结果是否符合预期，识别失败原因，决定重试策略。

输入：
- `ExecutionResult`（执行结果）
- `ExpectedOutput`（预期输出）
- `SuccessCriteria`（成功标准）

输出：
- `ValidationResult`：验证结果（通过/失败、差异分析）
- `FailureAnalysis`：失败原因分析
- `RetryStrategy`：重试建议（重试/调整参数/更换工具/人工介入）

建议实现：
- 基于 OP-09 的基础，增强为执行结果验证
- 验证类型：输出格式、数据完整性、业务逻辑、副作用确认
- 失败分类：超时、权限不足、工具错误、输入错误、环境问题
- 智能重试：根据失败类型决定是否重试、如何调整参数
- 降级策略：主工具失败时自动切换备选工具

验收标准：
- 文件写入后验证文件是否存在且内容正确
- LLM 输出验证格式是否符合要求
- 代码执行验证返回值是否符合预期
- 网络请求验证响应状态码和数据格式
- 失败时能给出明确的失败原因和修复建议

测试：
- 成功验证
- 格式错误检测
- 数据不完整检测
- 副作用验证
- 重试策略生成

#### OP-25 反思与策略优化

目标：深度分析任务执行过程，生成改进建议，更新记忆系统。

输入：
- 完整任务执行记录
- 工具使用情况
- 代码生成与执行记录
- 用户反馈

输出：
- `DeepReflection`：深度反思报告
- `StrategyOptimization`：策略优化建议
- `MemoryUpdateProposal`：记忆更新建议

建议实现：
- 基于 OP-10 的基础，增强为深度反思
- 分析维度：
  - 工具选择是否最优（是否有更快/更便宜的工具）
  - 代码质量如何（是否有更优雅的实现）
  - 执行效率如何（是否有并行优化空间）

## 5. 编程 AI 执行规则

4. 新增模块时补充测试
5. 运行 `conda run -n openpilot python -m pytest`
6. 不自动执行高风险动作
7. 不在日志、测试快照或文档中写入真实 secret

## 8. 风险与挑战

### 技术风险

1. **沙箱逃逸风险**
   - 缓解：使用成熟的沙箱方案（Docker、subprocess 隔离）
   - 缓解：代码静态分析检测危险操作
   - 缓解：资源限制（CPU、内存、网络）

2. **代码生成质量**
   - 缓解：多次生成取最优
   - 缓解：执行前代码审查
   - 缓解：执行后结果验证
   - 缓解：失败后自动修复

3. **执行失败处理**
   - 缓解：详细的错误日志
   - 缓解：智能重试策略
   - 缓解：备选工具/方案
   - 缓解：优雅降级到人工介入

### 产品风险

1. **用户信任问题**
   - 缓解：透明的执行过程展示
   - 缓解：完整的审计日志
   - 缓解：渐进式自主度提升
   - 缓解：用户随时可以干预

2. **过度自动化**
   - 缓解：高风险操作始终需要确认
   - 缓解：用户可以降低自主度
   - 缓解：异常情况主动请求人工介入

3. **学习偏差**
   - 缓解：用户可以查看和修改记忆
   - 缓解：置信度有上限（不会 100% 自动）
   - 缓解：定期提示用户复查偏好

### 应对策略

1. **分阶段推出**：
   - 第一阶段：只执行文件读取、LLM 调用等低风险工具
   - 第二阶段：增加文件写入等中风险工具
   - 第三阶段：增加代码执行能力
   - 第四阶段：增加网络请求、系统操作

2. **用户控制**：
   - 提供 `--no-execute` 模式（只规划不执行）
   - 提供 `--auto-approve` 模式（自动批准低风险操作）
   - 提供自主度设置（保守/平衡/激进）
   - 提供工具白名单/黑名单

3. **持续监控**：
   - 收集执行成功率数据
   - 收集用户反馈
   - 监控安全事件
   - 定期审查审计日志
